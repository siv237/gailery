# Рецензия: Ошибки в PIPELINE.md → путь к идеальному пайплайну

Дата рецензии: 2025-05-17
Рецензент: Cascade AI
Цель: привести проект к идеальному пайплайну (см. `05_идеальный_PIPELINE.md`)

---

## 1. КРИТИЧЕСКАЯ: Неоптимальный порядок шагов

**PIPELINE.md описывает:**
```
1. НАПОЛНЕНИЕ → 2. EXIF → 3. ОПИСАНИЕ → 4. ЛИЦА → 5. ИНДЕКСАЦИЯ
```

**Код pipeline.py выполняет:**
```
1. SCAN → 2. DESCRIBE → 3. FACES → 4. EXIF → 5. EMBED
```

**Идеальный порядок:**
```
1. НАПОЛНЕНИЕ → 2. EXIF → 3. ЛИЦА → 4. ОПИСАНИЕ → 5. ИНДЕКСАЦИЯ
```

**Обоснование:**
- **EXIF вторым** (без GPU) — пользователь сразу видит даты, GPS. Данные доступны для всех последующих шагов.
- **ЛИЦА третьим** (до описания!) — InsightFace ~50-100мс/фото, в 20-100× быстрее llama. Даёт имена персон + позиции лиц ДО описания. `faces_present` — ground truth от InsightFace, не угадывание.
- **ОПИСАНИЕ четвёртым** — llama-server получает имена + bbox → «Мария и Иван за столом» вместо «двое людей за столом».
- **ИНДЕКСАЦИЯ последним** — embed получает ВСЕ данные (описание с именами + EXIF + лица).

Ни PIPELINE.md, ни код не реализуют оптимальный порядок. Оба нуждаются в исправлении.

## 2. КРИТИЧЕСКАЯ: Отсутствует раздел SQLite-арбитраж

**PIPELINE.md** описывает GPU-арбитраж, но **полностью игнорирует SQLite-арбитраж**.

SQLite позволяет только одного писателя одновременно (даже в WAL mode). В проекте есть MQTT-механизм `db/cmd` → pipeline.py сериализует записи, но это нигде не документировано.

**Должно быть описано:**
- Pipeline.py — главный писатель (воркеры через subprocess.run, по очереди)
- API → MQTT очередь `db/cmd` → pipeline.py → `db/result`
- Сигнал `db/writing` — API знает когда БД занята
- Fallback — API пишет напрямую если pipeline остановлен
- WAL + busy_timeout=30с как страховка

## 3. КРИТИЧЕСКАЯ: Отсутствует каскад при изменении лиц

**PIPELINE.md** не описывает что происходит при:
- Переименовании персоны
- Объединении персон
- Ручном назначении лица другой персоне

При новом порядке (FACES → DESCRIBE → EMBED) описания содержат имена персон. Изменение имени → описание устарело → эмбеддинг устарел.

**Должен быть каскад:**
```
Изменение персоны → сброс description + embedded для всех фото с этой персоной
  → pipeline переопишет с новым именем → переиндексирует
```

## 4. СРЕДНЯЯ: Хеширование — НЕ отдельная фаза

PIPELINE.md описывает «Фаза B: Хеширование батчами» с `hash_limit=500`.

В реальности `scan_catalog.py` хеширует **инлайн при сканировании**. Отдельной фазы нет. Параметр `--hash-limit` не существует в pipeline.py.

**Решение:** Описать хеширование как часть единого прохода сканирования. Убрать `--hash-limit`.

---

## 5. СРЕДНЯЯ: ingest.py описан как воркер, но pipeline использует scan_catalog.py

Pipeline.py вызывает `scan_catalog.py --scan` (скан + hash + dedup + ingest в одном). ingest.py — только для ручного запуска через UI кнопку.

**Решение:** Указать что в pipeline наполнение = `scan_catalog.py --scan`.

## 6. МЕЛКАЯ: Флаг `hash` в списке, но не создаётся

scan_catalog.py не создаёт файл-флаг `hash`. Убрать из списка.

## 7. МЕЛКАЯ: Неточные счётчики прогресса

- Описание «найдено/уникальных» не соответствует реальному «ingested/canonical»
- Нет фильтра `content_hash IS NOT NULL`
- Счётчик лиц должен учитывать новый порядок: `faces_done=1 / всего` (а не `faces в БД / faces_present=1`)

**Решение:** Переписать формулы счётчиков по 05_идеальный_PIPELINE.md.

## 8. МЕЛКАЯ: Видео-EXIF и видео-обработка не описаны

- Видео получают `media_type='video'` при наполнении
- EXIF видео (ffprobe) — в коде есть, но недоступен для новых видео (баг в exif.py)
- Описание видео: автоматически `[видео]` без llama
- Describe/Faces/Embed НЕ обрабатывают видео

Всё это должно быть в PIPELINE.md.
