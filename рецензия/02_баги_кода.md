## Баги, БЛОКИРУЮЩИЕ переход к идеальному пайплайну

### 1. [БЛОКЕР] Порядок шагов в pipeline.py — FACES после DESCRIBE

**Файл:** `pipeline.py`, строки 523-543

Текущий порядок: SCAN → DESCRIBE → FACES → EXIF → EMBED.
Идеальный порядок: SCAN → EXIF → **FACES → DESCRIBE** → EMBED.

**Что менять:**
- Переставить вызов `run_step("FACES", ...)` ПЕРЕД `run_step("DESCRIBE", ...)`
- Переставить вызов `run_step("EXIF", ...)` сразу после SCAN
- faces.py: убрать зависимость от `faces_present=1`, сканировать ВСЕ фото
- describe.py: получать имена персон и bbox из faces таблицы, передавать в промпт llama

### 2. [БЛОКЕР] faces.py зависит от faces_present (ставится describe)

**Файл:** `faces.py`

Сейчас faces.py выбирает фото с `faces_present=1`. Этот флаг ставит describe.py. При новом порядке (FACES до DESCRIBE) describe ещё не запускался → `faces_present=0` у всех → faces.py ничего не обработает.

**Исправление:** faces.py должен выбирать ВСЕ canonical фото с `faces_done=0`, без фильтра `faces_present`. InsightFace сам определит есть ли лица. После обработки:
- Лица найдены → `faces_present=1, faces_done=1`
- Лиц нет → `faces_present=0, faces_done=1`

### 3. [БЛОКЕР] describe.py не передаёт имена лиц в промпт

**Файл:** `describe.py`

Сейчас llama описывает фото «вслепую» — без информации о лицах. При новом порядке к моменту describe уже есть faces + personas.

**Нужно добавить:**
- Запрос лиц и персон для каждого фото перед описанием
- Формирование контекста: количество лиц, имена, позиции (из bbox)
- Включение контекста в промпт:
```
На фото обнаружено 2 лица:
- Мария (слева)
- Иван (справа)
Опиши что на фото, используя имена людей.
```

### 4. [БЛОКЕР] Нет каскада при изменении персоны

**Файл:** `src/api/persons.py`

При `update_persona` (переименование) **не вызывается** `invalidate_embeddings_for_persona`. Описания и эмбеддинги с устаревшим именем остаются.

При `merge_personas` вызывается `invalidate_embeddings_for_persona` → сброс embedded, но **не сброс description**. Описание содержит старое имя → embed получит старый текст.

**Нужный каскад (для rename/merge/delete/assign):**
```python
# Для каждого фото с лицами этой персоны:
description = NULL     # переописать с новым именем
described = 0
embedded = 0           # переиндексировать
DELETE from LanceDB    # удалить старый вектор
```

## Баги, существующие в текущем коде

### 5. [КРИТИЧЕСКИЙ] describe.py — NameError в Ollama-режиме

**Файл:** `describe.py`, функция `_main_ollama()`

Переменные `described` и `failed` не инициализированы. Ollama-режим describe полностью нерабочий.

**Исправление:** Добавить `described = 0; failed = 0` перед циклом.

### 6. [КРИТИЧЕСКИЙ] exif.py — видео исключены из EXIF-обработки

**Файл:** `exif.py`

SQL-фильтр `media_type != 'video'` исключает видео. Код обработки видео-метаданных есть, но недостижим.

**Исправление:** Убрать фильтр, или сделать отдельный проход для видео.

### 7. [СРЕДНИЙ] scan_catalog.py — stale-эмбеддинги не удаляются из LanceDB

**Файл:** `scan_catalog.py`, `_mark_stale()`

Флаг `embedded=0` ставится, но старый вектор в LanceDB остаётся → мусор и искажение поиска.

**Исправление:** Добавить `db.delete_photo_embedding(photo_id)` при stale.

### 8. [СРЕДНИЙ] Несогласованные наборы видео-расширений

Три разных набора в scan_catalog.py, _ingest_new_canonical, pipeline.py get_progress. Расширения `.mpg .mpeg .m4v .flv .vob .ts` не попадут в каталог.

**Исправление:** Единый набор `VIDEO_EXTS` в config.py, использовать везде.

### 9. [МЕЛКИЙ] scan_catalog.py — мёртвые верхнерегистровые расширения

`SUPPORTED_EXTS` содержит `.MP4`, `.MOV` и т.д., но `Path(fn).suffix.lower()` всегда возвращает нижний регистр. Мёртвый код.

### 10. [МЕЛКИЙ] exif.py — N+1 запрос для canonical проверки

Фильтрация canonical через Python (`is_path_canonical()`) вместо SQL JOIN. Неэффективно.

### 11. [МЕЛКИЙ] faces.py — legacy photo_id (относительный путь)

faces.photo_id хранит относительный путь вместо UUID. Legacy-поведение, создающее путаницу. Связь с photos должна идти через content_hash.
