# Gailery — Пайплайн обработки фото

## Принцип

Пайплайн — бесконечный цикл последовательных шагов.
Каждый шаг запускает одного воркера и ждёт его завершения.
Между шагами закрываем подключение к БД.

Один GPU — один процесс. Никаких фоновых процессов. Никакого параллелизма.

---

## Порядок шагов

1. НАПОЛНЕНИЕ — сбор путей + хеширование батчами + дедуп + ingest
2. EXIF — чтение метаданных (не GPU)
3. ЛИЦА — InsightFace находит и кластеризует лица (GPU)
4. ОПИСАНИЕ — VLM описывает фото с именами лиц (GPU)
5. ИНДЕКСАЦИЯ — семантические эмбеддинги для поиска (GPU)

Порядок Лица→Описание: VLM получает имена персон из шага Лица и использует их в описании.

AI-шаги работают батчевым циклом: faces(60) → describe(60) → embed(60) → повтор пока не 100%.

---

## Шаг 1: НАПОЛНЕНИЕ

### Фаза A: Сбор путей (`scan_catalog.py --scan`)

1. Берём список корневых каталогов (включённых из `catalog_roots`)
2. Обходим каждый каталог рекурсивно
3. Каждый файл с подходящим расширением — проверяем: этот путь уже есть в БД?
   - Нет → добавляем запись (путь, размер, дата изменения), хеш пустой
   - Да, и файл не менялся → пропускаем
   - Да, и файл изменился (размер или дата) → обновляем размер/дату, при необходимости пересчитаем хеш
4. Если файл пропал с диска — помечаем `deleted=1, deleted_type='auto_missing'`
5. Если файл вернулся и хеш совпадает — снимаем `deleted=0`, флаги обработки НЕ сбрасываем
6. При повторном скане — каталоги где ничего не менялось (по mtime < scanned_at) пропускаем целиком

Результат: в БД все пути файлов, готовые к хешированию. ~1с для 8680 файлов.

### Фаза B: Хеширование батчами (`scan_catalog.py --hash --limit N`)

1. Выбираем canonical файлы у которых хеш ещё не посчитан (content_hash пустой, не удалён)
2. Берём не больше N файлов за раз (параметр hash_limit, по умолчанию 200)
3. Для каждого файла: считаем xxh128 хеш содержимого
4. Записываем хеш в БД батчами (каждые 200 файлов)
5. Логируем прогресс каждые 200 файлов: «Хеширование 1200/8680 (45с, 8/с)»

Скорость: ~8 файлов/с, упирается в диск. 200 файлов = ~25с.
Память pipeline: ~140MB при любом лимите (было 6.6GB при лимите 500).

Результат: у файлов есть уникальные хеши, можно определить дубли

### Фаза C: Дедупликация + Наполнение photos (`scan_catalog.py --dedup-ingest`)

1. Группируем файлы по content_hash
2. В каждой группе: один файл = canonical (предпочитаем уже описанный, потом с кратчайшим путём)
3. Остальные файлы группы = не-canonical, у них дописываем путь к существующему хешу
4. Canonical файлы у которых ещё нет записи в photos — добавляем
5. Пометить `ingested=1`

Результат: в photos только уникальные canonical фото, готовые к обработке

### Цикл наполнения в pipeline.py

```
Пока unhashed > 0 ИЛИ exif < 100%:
  СКАН (пути)
  Если unhashed > 0:
    ХЕШ (hash_limit)
    ДЕДУП+INGEST
  Если exif < 100%:
    EXIF (--all)
```

### Счётчик Наполнение

- canonical_total = все canonical файлы (с хешем И без)
- canonical_hashed = canonical с content_hash
- unhashed = canonical_total - canonical_hashed
- Наполнение 100% ТОЛЬКО когда unhashed=0 (все хеши посчитаны)

## Шаг 2: EXIF (`exif.py --all`)

1. Находим canonical фото с хешем, у которых EXIF ещё не читали
2. Читаем EXIF: дата, GPS, камера (exifread для фото, ffprobe для видео)
3. Помечаем exif_checked=1 даже если EXIF нет (чтобы не повторять)
4. Всегда `--all` — сколько в photos, столько и обработаем
5. Скорость: ~27 фото/с, 8 потоков

Не требует GPU — можно запускать пока видеокарта свободна

---

## Шаг 3: ЛИЦА (`faces.py --limit 60`)

1. Убиваем сирот llama-server (ppid=1)
2. Находим canonical фото с хешем (не видео), у которых `faces_done=0`
3. InsightFace: детекция лиц, векторные представления
4. DBSCAN кластеризация векторов → персоны (авто-имена cluster_*)
5. Записываем лица в БД с content_hash
6. Помечаем `faces_done=1`; с лицами: `faces_present=1`; без лиц: `faces_present=0`
7. Лимит: N фото за раз (по умолчанию 60, 0 = все)

GPU занят

---

## Шаг 4: ОПИСАНИЕ (`describe.py --limit 60 --batch-size 6`)

1. Убиваем сирот llama-server
2. Находим canonical фото с хешем, у которых нет описания
3. Видеофайлы → сразу ставим описание «[видео]» без VLM
4. Запускаем Ollama VLM модель
5. VLM генерирует текстовое описание + определяет `has_faces`
6. Если `faces_done=1` — VLM НЕ перетирает `faces_present`
7. Контекст в промте:
   - **Лица**: имена персон + позиции (слева/центр/справа) + комментарии персон
   - **Дата съёмки**: manual_date или date из EXIF
   - **Папка**: alias корневого каталога
8. Лимит: N описаний за раз (по умолчанию 60), батч VLM (по умолчанию 6)

GPU занят всё время работы VLM

---

## Шаг 5: ИНДЕКСАЦИЯ (`embed.py --limit 60`)

1. Убиваем сирот llama-server
2. Находим canonical фото с хешем и описанием, у которых нет эмбеддинга
3. Собираем текст: описание + имена лиц + папка + дата
4. Генерируем векторные эмбеддинги, храним в LanceDB
5. Лимит: N фото за раз (по умолчанию 60, 0 = все)

GPU занят

---

## Цикл pipeline.py

```
Пока не остановлен:
  1. Обработать MQTT команды от API
  2. Собрать системные метрики
  3. Посчитать прогресс всех шагов
  4. Если все 100% — спать 30с, ждать сигнала проснуться

  === Этап 1: Наполнение ===
  5. СКАН (пути)
  6. Пока unhashed > 0 ИЛИ exif < 100%:
       Если unhashed > 0: ХЕШ(hash_limit) → ДЕДУП+INGEST
       Если exif < 100%: EXIF(--all)

  === Этап 2: AI батчевый цикл ===
  7. Если faces < 100%: FACES(--limit 60)
  8. Если describe < 100%: DESCRIBE(--limit 60 --batch-size 6)
  9. Если embed < 100%: EMBED(--limit 60)
  10. Оптимизация LanceDB (дедуп + компакт)
```

---

## Параметры pipeline.py

| Параметр | По умолчанию | Что делает |
|---|---|---|
| --batch | 60 | Общий размер батча (AI-шаги) |
| --ingest | 0 | Переопределить батч наполнения (0 = --batch) |
| --describe | 0 | Переопределить батч описания (0 = --batch) |
| --batch-size | 6 | Батч VLM (параллельных запросов) |
| --hash-limit | 200 | Файлов хешировать за раз (0 = все) |
| --root | "" | Ограничить по конкретному root_id |

Каждый шаг можно запустить отдельно:

| Шаг | Параметр | По умолчанию | Воркер |
|---|---|---|---|
| Скан каталога | — | — | scan_catalog.py --scan |

| Описание | desc_limit | 60 | describe.py --limit N --batch-size M |

## Контекст описания фото

VLM получает помимо изображения текстовый контекст:

1. **Лица**: запрос `faces` + `personas` по content_hash
   - Имя персоны + позиция на фото (слева/центр/справа)
   - Комментарий персоны (если есть)
   - Пример: `"На фото обнаружены лица: Анна (слева, мама), Пётр (в центре). Также 1 лиц без имён. Используй имена в описании если они подходят к людям на фото."`
2. **Дата съёмки**: manual_date или date из photos
3. **Папка**: alias из catalog_roots через catalog_files

## MQTT

### GPU арбитраж

- Топик gpu/lock (retained): кто держит GPU, с какого времени, какой PID
- Фоновые воркеры: acquire_gpu() — мьютекс, ждёт если занят
- Временные задачи (поиск, обогащение): мягкий или жёсткий захват

### Pipeline ↔ API

- Pipeline подписан на топик db/cmd — получает команды от API
- Выполняет: обновить фото, установить GPS, сбросить шаг, объединить персоны и т.д.
- Отвечает на топик db/result/{request_id}

### Статус

- Флаг-файлы в FLAG_DIR: pipeline, describe, faces, exif, embed, hash, pipeline_idle
- Pipeline публикует текущий шаг в MQTT → API показывает в UI

## Watchdog

Пёс отвечает ТОЛЬКО за pipeline. Ничего больше.

| Событие | Реакция пса |
|---|---|
| Кнопка «Цепочка» | Просыпается: убирает no_restart, включает pipeline |
| Кнопка «Стоп» | Засыпает: ставит no_restart, выключает pipeline |
| Pipeline упал (пёс активен) | Перезапускает |
| Сироты llama-server (пёс активен) | Убивает |
| Спящий пёс | Ничего не делает |

## Устаревание результатов

Если у canonical файла изменился content_hash (файл перезаписан):

1. Описание → стереть, faces_present=0
2. Записи лиц с этим хешем → удалить
3. Эмбеддинг → удалить из LanceDB, embedded=0
4. EXIF → exif_checked=0

Гарантирует что результаты всегда соответствуют содержимому файла

## Счётчики прогресса

Все считаются только по canonical файлам с хешем (уникальным):

| Шаг | Сколько сделано | Из скольких |
|---|---|---|
| Наполнение | canonical+hashed в photos | canonical всего (с хешем и без) |
| unhashed | canonical без хеша (отдельный счётчик) | — |
| EXIF | canonical+hashed с exif | canonical+hashed фото |
| Лица | canonical+hashed с faces_done=1 | canonical+hashed (не видео) |
| Описание | canonical+hashed с описанием | canonical+hashed (не видео) |
| Индексация | canonical+hashed с эмбеддингом | canonical+hashed (не видео) с описанием |

canonical+hashed = is_canonical=1 AND content_hash IS NOT NULL AND deleted=0

## Логирование

Все воркеры пишут в logs/pipeline.log через log() (НЕ print!)
API читает этот файл для /api/log

Формат: [дата-время] [ТЕГ] сообщение

Теги: CATALOG, DESCRIBE, FACES, EXIF, EMBED, PIPELINE, ENRICH, WATCHDOG, CONTROL

Хеширование: лог каждые 200 файлов с процентом и скоростью
