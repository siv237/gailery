# Gailery — Пайплайн обработки фото

## Принцип

Пайплайн — бесконечный цикл последовательных шагов.
Каждый шаг запускает одного воркера и ждёт его завершения.
Между шагами закрываем подключение к БД.

Один GPU — один процесс. Никаких фоновых процессов. Никакого параллелизма.

---

## Порядок шагов

1. НАПОЛНЕНИЕ — сбор путей + хеширование + добавление в photos
2. EXIF — чтение метаданных (не GPU)
3. ОПИСАНИЕ — VLM описывает фото (GPU)
4. ЛИЦА — InsightFace находит и кластеризует лица (GPU)
5. ИНДЕКСАЦИЯ — семантические эмбеддинги для поиска (GPU)

---

## Шаг 1: НАПОЛНЕНИЕ

### Фаза A: Сбор путей

1. Берём список корневых каталогов (включённых)
2. Обходим каждый каталог рекурсивно
3. Каждый файл с подходящим расширением — проверяем: этот путь уже есть в БД?
   - Нет → добавляем запись (путь, размер, дата изменения), хеш пустой
   - Да, и файл не менялся → пропускаем
   - Да, и файл изменился (размер или дата) → обновляем размер/дату, при необходимости пересчитаем хеш
4. Если файл пропал с диска — помечаем удалённым
5. Если файл вернулся — снимаем пометку удаления
6. При повторном скане — каталоги где ничего не менялось (по дате изменения папки) пропускаем целиком
7. Виджет показывает сколько уникальных путей найдено — цифра растёт в реальном времени

Результат: в БД все пути файлов, готовые к хешированию

### Фаза B: Хеширование батчами

1. Выбираем файлы у которых хеш ещё не посчитан (content_hash пустой, размер > 0, не удалён)
2. Берём не больше N файлов за раз (параметр hash_limit, по умолчанию 500)
3. Для каждого файла: считаем xxh128 хеш содержимого
4. Записываем хеш в БД батчами (каждые 500 файлов)
5. Логируем прогресс каждые 200 файлов: «Хеширование 1200/180000 (45с, 26/с)»

Результат: у файлов есть уникальные хеши, можно определить дубли

### Фаза C: Дедупликация + Наполнение photos

1. Группируем файлы по content_hash
2. В каждой группе: один файл = canonical (предпочитаем уже описанный, потом с кратчайшим путём)
3. Остальные файлы группы = не-canonical, у них дописываем путь к существующему хешу
4. Canonical файлы у которых ещё нет записи в photos — добавляем
   - Проверяем что фото с таким путём уже не существует (не добавляем повторно)
5. Удаляем из photos дубли (несколько записей с одним путём)

Результат: в photos только уникальные canonical фото, готовые к обработке

### Счётчик Наполнение

- Найдено = все пути файлов в БД (не удалённые)
- Уникальных = canonical файлы с хешем (не удалённые)
- Прогресс = уникальных / найденных
- По мере нахождения дублей процент может снижаться — это нормально

### Важно

- Файлы без хеша — нормальные записи, просто ещё не обработаны
- Все downstream воркеры работают ТОЛЬКО с canonical файлами у которых есть хеш
- Счётчики downstream считают от canonical+hashed, не от общего числа

---

## Шаг 2: EXIF

1. Находим canonical фото с хешем, у которых EXIF ещё не читали
2. Читаем EXIF: дата, GPS, камера
3. Помечаем exif_checked=1 даже если EXIF нет (чтобы не повторять)
4. Лимит: N фото за раз (по умолчанию 500)

Не требует GPU — можно запускать пока видеокарта свободна

---

## Шаг 3: ОПИСАНИЕ

1. Убиваем сирот llama-server (процессы у которых родитель умер)
2. Находим canonical фото с хешем, у которых нет описания
3. Видеофайлы → сразу ставим описание «[видео]» без VLM
4. Запускаем llama-server + VLM модель
5. VLM генерирует текстовое описание + определяет есть ли лица (faces_present)
6. Закрываем llama-server
7. Лимит: N описаний за раз (по умолчанию 60), батч VLM (по умолчанию 6)

GPU занят всё время работы VLM

---

## Шаг 4: ЛИЦА

1. Убиваем сирот llama-server
2. Находим canonical фото с хешем, где VLM обнаружил лица, но ещё не нашли InsightFace
3. InsightFace: детекция лиц, векторные представления
4. DBSCAN кластеризация векторов → персоны
5. Записываем лица в БД с content_hash
6. Лимит: N фото за раз (0 = все)

GPU занят

---

## Шаг 5: ИНДЕКСАЦИЯ

1. Убиваем сирот llama-server
2. Находим canonical фото с хешем и описанием, у которых нет эмбеддинга
3. Собираем текст: описание + имена лиц + папка + дата
4. Запускаем llama-server + embedding модель
5. Генерируем векторные эмбеддинги, храним в LanceDB
6. Закрываем llama-server
7. Лимит: N фото за раз (0 = все)

GPU занят

---

## Цикл pipeline.py

```
Пока не остановлен:
  1. Обработать MQTT команды от API
  2. Собрать системные метрики
  3. Посчитать прогресс всех шагов
  4. ЗАПУСК: СКАН (сбор путей)
  5. Если есть файлы без хеша:
       ЗАПУСК: ХЕШИРОВАНИЕ (с лимитом N)
  6. ЗАПУСК: EXIF (с лимитом N)
  7. ЗАПУСК: ОПИСАНИЕ (с лимитом N, батч M)
  8. ЗАПУСК: ЛИЦА (с лимитом N)
  9. ЗАПУСК: ИНДЕКСАЦИЯ (с лимитом N)
  10. Оптимизация LanceDB (дедуп + компакт)
  11. Если прогресса нет — спать 30с, ждать сигнала проснуться
  12. Если все 100% — спать 30с, ждать новых фото
```

## Параметры pipeline.py

| Параметр | По умолчанию | Что делает |
|---|---|---|
| --batch | 60 | Общий размер батча |
| --ingest | 0 | Переопределить батч наполнения (0 = --batch) |
| --describe | 0 | Переопределить батч описания (0 = --batch) |
| --batch-size | 6 | Батч VLM |
| --hash-limit | 500 | Файлов хешировать за раз (0 = все) |
| --root | "" | Ограничить по конкретному корневому каталогу |

## MQTT

### GPU арбитраж

- Топик gpu/lock (retained): кто держит GPU, с какого времени, какой PID
- Фоновые воркеры: acquire_gpu() — мьютекс, ждёт если занят
- Временные задачи (поиск, обогащение): мягкий или жёсткий захват

### Pipeline ↔ API

- Pipeline подписан на топик db/cmd — получает команды от API
- Выполняет: обновить фото, установить GPS, сбросить шаг, объединить персоны и т.д.
- Отвечает на топик db/result/{request_id}

### Статус

- Флаг-файлы в FLAG_DIR: pipeline, describe, faces, exif, embed, hash, pipeline_idle
- Pipeline публикует текущий шаг в MQTT → API показывает в UI

---

## Watchdog

Пёс отвечает ТОЛЬКО за pipeline. Ничего больше.

| Событие | Реакция пса |
|---|---|
| Кнопка «Цепочка» | Просыпается: убирает no_restart, включает pipeline |
| Кнопка «Стоп» | Засыпает: ставит no_restart, выключает pipeline |
| Pipeline упал (пёс активен) | Перезапускает |
| Сироты llama-server (пёс активен) | Убивает |
| Спящий пёс | Ничего не делает |

## Устаревание результатов

Если у canonical файла изменился content_hash (файл перезаписан):

1. Описание → стереть, faces_present=0
2. Записи лиц с этим хешем → удалить
3. Эмбеддинг → удалить из LanceDB, embedded=0
4. EXIF → exif_checked=0

Гарантирует что результаты всегда соответствуют содержимому файла

---

## Счётчики прогресса

Все считаются только по canonical файлам с хешем (уникальным):

| Шаг | Сколько сделано | Из скольких |
|---|---|---|
| Наполнение | canonical+hashed в photos | canonical+hashed всего |
| EXIF | canonical+hashed с exif | canonical+hashed фото |
| Описание | canonical+hashed с описанием | canonical+hashed фото |
| Лица | canonical+hashed с найденными лицами | canonical+hashed где VLM обнаружил лица |
| Индексация | canonical+hashed с эмбеддингом | canonical+hashed фото |

canonical+hashed = is_canonical=1 AND content_hash не пустой AND deleted=0

## Логирование

Все воркеры пишут в logs/pipeline.log через log() (НЕ print!)
API читает этот файл для /api/log

Формат: [дата-время] [ТЕГ] сообщение

Теги: CATALOG, DESCRIBE, FACES, EXIF, EMBED, PIPELINE, ENRICH, WATCHDOG, CONTROL

Хеширование: лог каждые 200 файлов с процентом и скоростью
